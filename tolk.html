Tolk — restructuring a feature-first language app around one core intent.

An early-stage, in-progress product. This case study focuses on the design decisions and the reasoning behind them — which is where most of the work lived.

The starting point

I'd built Tolk feature-first: a word search, a music-lyrics mode, flashcards, a quiz, grammar, settings, and history — all surfaced flat in a five-item bottom navigation. Each feature worked in isolation, but the product had no spine. My first planning artifact was a single diagram trying to be three things at once — a user flow, a sitemap, and a data model — with shapes and arrows that meant different things in different places.

So the real problem wasn't "add features." It was: what is this product, what's the one thing a user comes here to do, and how should everything else organize around that?

How I approached it

I worked validation-first and decision-led, not feature-led. Three principles drove the work:

  • Separate the concerns

    A flow, a sitemap, and a content model are three documents with three conventions. Collapsing them is what made the original unreadable — so I split them apart, each with one shape language and one meaning per arrow.

  • Build only what's validated

    Where a feature implied real infrastructure (accounts, a backend), I looked for the cheapest way to test the underlying assumption before committing engineering to it.

  • Match complexity to the user

    Depth should scale with the learner's level and the device's space — not show everything to everyone everywhere.

Decision 01 · Information architecture

Navigation by intent: five tabs down to four

The old bottom bar mixed unrelated mental modes — a lookup tool, passive learning, active practice, reference, and configuration — competing for the same scarce space. I regrouped navigation around user intent and cut it to four: Search, Games, Grammar, Profile. Settings, low-frequency, moved into Profile; an underdeveloped Music mode was deferred rather than shipped half-built.

Shows: information architecture, mental models, editing down.

Sitemap — the restructured four-tab architecture, organized by intent.

Decision 02 · Prioritization

Protecting activation: Search stays the landing screen

The original structure proposed a summary dashboard as home. I pushed back: Tolk is dictionary-first, so the highest-value move is to land the user directly in the core action with zero friction — that serves activation. A progress dashboard serves retention, which wasn't validated yet, so I moved those stats into Profile rather than letting them gate the main task. I documented a lighter option — a thin progress strip above Search — as a future move if retention becomes the priority.

Shows: separating activation from retention, resisting vanity surfaces, sequencing by what's validated.

Decision 03 · Flow rigor

Resolving the onboarding flow

Onboarding branched on the user's level — beginner, advanced, or "not sure" — but the original flow never showed how "not sure" resolved. I closed the gap: the placement test routes into the same two outcomes as the explicit choices, and each path sets both the interface language and the level (beginner → native-language UI; advanced → German UI). I also removed the account-creation step, because it contradicted a later decision about authentication.

Shows: flow rigor, handling the branch everyone forgets, internal consistency.

Onboarding flow — the "not sure" path resolves into the same two outcomes.

Decision 04 · Scope discipline

Deferring accounts: local-first persistence

"Save my progress" reads as "build login," but they're not the same thing. I separated them: progress persists locally on the device first, with no account required. This keeps the product feeling like it remembers the user while a lightweight email-capture test measures whether people actually want to register — before committing to building authentication. I structured the saved data as a single versioned store, so that once registration intent is validated, migrating local progress into real accounts is a clean one-step job rather than a rewrite.

Shows: scope discipline, validation-driven design, designing for a future state without over-building the present.

Decision 05 · Progressive disclosure

Level-adaptive conjugation, on a real content model

German verbs carry a lot of conjugation. Showing all nine tenses to a beginner is noise; hiding them from an advanced learner is a gap. I tied conjugation depth to the user's level — a compact everyday set for beginners, the full set for advanced — and designed the display to adapt per device. Underneath sits a structured content model: every word entry resolves to an image, gender, meaning, a sample sentence, and a grammatical type that branches into verb, noun, adjective, or connector — each carrying only the depth that type needs.

Shows: progressive disclosure, content-dense UX, a real linguistic data model.

Content model — one entry, typed by part of speech, with depth scaled to type.

Decision 06 · Responsive system

One consistent pattern across devices

I extended the mobile-first app to iPad and laptop with a hybrid model — most views stay a centered, readable column; a few (word entry, conjugation, games, stats) use the extra width — while keeping one navigation pattern across devices so users never relearn the layout. Along the way I corrected a stats block that had shipped unstyled, rebuilding it so the number is the visual hero and the label recedes, making progress legible at a glance.

Shows: responsive / cross-device design, design-system consistency, visual hierarchy.

Desktop — the word entry opens into a two-column layout, with the conjugation grid showing multiple tenses at once.

What this project shows

  • Turning an ambiguous, feature-first product into a coherent structure organized around a single core intent.

  • Information architecture and user-flow design — including the edge cases that usually get skipped — plus prioritization and the discipline to decide what not to build yet, and why.

  • Designing for validation (cheap signals before expensive infrastructure), progressive disclosure over a real content model, and a responsive system kept consistent by design-system thinking — every choice paired with its rationale and trade-offs.

Status & next steps

The restructured app is a working prototype, live now. Next: validate registration intent via the email-capture test, expand content, and — if the signal is there — layer in accounts, migrating the local progress already designed for it. A longer-term path moves the app toward the iOS App Store.
